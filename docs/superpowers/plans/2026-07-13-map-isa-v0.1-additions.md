# Plan — MAP ISA v0.1: the additions the benchmark suite demanded

**Date:** 2026-07-13
**Spec:** [Benchmark suite design](../specs/2026-07-13-benchmark-suite-design.md) ·
[Coverage audit](../../../benchmarks/coverage.md)
**Status:** In progress.

Three additions, in the order the ladder demands them. Each is a full vertical
(Sail → emulator → ISS → IR → lowering → RTL → cosim → eDSL), and each closes
exactly one benchmark. Nothing else changes: **PP v0 stays frozen** — it
survived its own suite.

The dev-phase contract is mutable, so encodings may be revised; the `[ir-breaking]`
commit hatch covers proto changes while all consumers are in-repo.

## Task 1 — reg-reg ALU (closes E3)

Demanded by exactly one program, which appears in **two independent corpora**
(p4 `calc`, xISA `network-calculator`), and which Nanuk fails outright: the
whole MAP ALU is `ADDI`/`ANDI`/`SHLI`, immediate-only. A machine that cannot
compute `a - b` where both operands came off the wire is not a compute machine.

### Encoding (frozen here)

Five new opcodes. The opcode field is 6 bits at [31:26] with 15 of 64 used, so
flat opcodes stay affordable and decode stays uniform — no funct field, no
second decode level.

| Mnemonic | Opcode | Semantics |
|---|---|---|
| `add rd, rs, rt` | `0x10` | `rd = rs + rt` (64-bit wraparound) |
| `sub rd, rs, rt` | `0x11` | `rd = rs - rt` (64-bit wraparound) |
| `and rd, rs, rt` | `0x12` | `rd = rs & rt` |
| `or  rd, rs, rt` | `0x13` | `rd = rs \| rt` |
| `xor rd, rs, rt` | `0x14` | `rd = rs ^ rt` |

Field layout follows `ADDI`/`BEQ` exactly:

```
[31:26] opcode   [25:23] rd   [22:20] rs   [19:17] rt   [16:0] reserved, zero
```

Reserved bits are enforced (the ISS already rejects non-zero reserved fields;
the decoder fuzz test will cover the new space).

### Deliberately NOT added

- **`MUL`.** xISA's calculator implements multiply as a shift-and-add *routine*,
  not an instruction, and Nanuk's 64-bit GPRs delete the entire carry-propagation
  dance xISA needs (it hand-builds a 64-bit accumulator out of 32-bit words).
- **Register-indirect branch.** xISA dispatches the calculator's opcode through
  a table whose values are *jump addresses*. A `BEQ` chain covers the behavior
  (12 instructions vs. 3). The *lesson* — that the control plane can redefine
  the data plane's control flow — is genuinely lost, and this is the one place
  Nanuk lands on "impossible" rather than "verbose". Named, not fixed.
- **Shift-by-register**, **ordered compares** (`BLT`/`BGE`), **right shift**.
  No corpus program forces them; the boundary rule cuts them.

### Known pressure (parameters, not ISA)

The audit predicted a 32-bit shift-and-add multiply runs ~224 steps against a
**256-step budget** and wants 6 live values against **4 GPRs**. Decide when
writing the calculator: either raise the budget (it is a parameter) or spec the
calculator with 16-bit operands. Failure mode would be `ERR_STEP_BUDGET`, which
reads like a bug rather than a limit — so this must not be discovered at
runtime.

### Steps

1. Sail: `types.sail` (union + encoding comment), `decode.sail` (encdec),
   `insts.sail` (execute). Rebuild the emulator.
2. Sail test: golden decode words in `spec/sail/test/map/`.
3. `map_encoding.py` (`OP_ADD`…`OP_XOR`, `encode_alu`), `map_asm.py` (mnemonics).
4. `map_iss.py`: decode + execute + trace.
5. IR: `nanuk_ir.proto` gains a reg-reg ALU op (`[ir-breaking]`), plus
   `map_validate` / `map_lower` / `map_interp` — the interpreter's step
   accounting must stay instruction-for-instruction with the lowering, or the
   differential tests lose their teeth.
6. eDSL: `nanuk.lang.match_action`.
7. RTL: `hw/amaranth/nanuk_amaranth/map.py`, then cosim.
8. `examples/calc` — the E3 benchmark, with acceptance vectors. **This is the
   point of the task**; the ISA change is in service of it.

## Task 2 — counter tables (closes T2)

Demanded by xISA `ipv4-counters`, p4 `p4runtime`, xdp `basic03/04`.

**The byte delta is implicit**: `packets += 1; bytes += plen`. xISA needs a
`SIZEQUERY` instruction only because *its* `COUNTER` takes an explicit delta
register; making ours implicit means **no program in any corpus ever reads the
frame length**, so no length instruction and no second system metadata slot.
The counter hardware consumes `plen`, which the core already knows from
streaming; the instruction set never exposes it.

This is Nanuk's **first data-plane table write**. The boundary against
data-plane *learning* (refused, parked) must be named explicitly in the spec: a
counter increments a value at an index the program computes; learning inserts a
*key*. Different capability, different machine.

Open sub-question to settle in the ISA doc: counter tables are a new table
*kind*, so the table-config record grows a type field, and `LOOKUP` must reject
counter tables (and the counter instruction must reject match tables) — totality
demands both.

## Task 3 — LPM tables (closes T3)

Demanded by 8 p4 tables, xdp's `bpf_fib_lookup`, and 3 of 5 xISA examples. The
single highest-value addition.

Keys fit **64 bits** (xISA's widest is a 44-bit VRF++DIP), so no multi-word-key
machinery follows. `LOOKUP`'s existing shape — key register, fused
branch-on-miss, action data in `rd` — carries over unchanged; only the match
semantics differ, which means this is a *table-kind* change rather than an
instruction change.

## Spec debt to close en route

**`LOOKUP`'s hit path is documented but under-specified in prose.** The Sail is
unambiguous (`insts.sail:125`: hit → `rd` = action data, fall through; miss →
`rd = 0`, branch), but the corpora demand per-entry *action selection*, not just
action data — p4's `calc` dispatches one table to five different action bodies.
Nanuk's answer is to pack an action id into the action word and `BEQ`-chain on
it. That convention needs writing down and demonstrating in `examples/calc`,
which Task 1 delivers anyway.

## Also worth writing down (no ISA change)

- **The headroom is legal read/write scratch.** `st` then `ld` at
  `h_frame, -8`; at `send 0` the drain starts at index 32, so it is never
  transmitted. This synthesizes a right shift, a `CONCAT`, and a sign-bit
  extract. Three separate audits leaned on it. It is an *idiom*, and it should
  be documented as one.
- **A PP→md header-present convention.** MAP cannot test header presence: a
  `LD` from an absent base is a fatal error halt. `map_ttl` turns that into a
  virtue ("totality is the guard") — which works exactly as long as the
  absent-header action is *drop*. `ipv4-counters` breaks it: it must **count**
  the non-IPv4 packet and *then* drop. The fix needs no ISA change (PP `STMD`s
  a present-bitmap; MAP `LDMD` + `ANDI` + `BEQ`), but it must become a
  documented convention, because today the only sanctioned answer is "die".
